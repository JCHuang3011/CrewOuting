把散落的工作文件
變成一個會幫你思考的知識庫

用一套有效率的方式，把你散落各處的工作資料整理成私有知識庫，
並透過它讓 AI 更精準地協助、分析你的工作。

它不是取代人的判斷，而是把資料整理成 AI 能反覆使用的工作記憶。

先問一個問題

你的工作知識，現在散落在哪裡？

🗂️到處都是檔案

共用磁碟、桌面、Email、Excel、PDF、會議記錄、聊天訊息……同一件事散在十個地方。

「上次那份 Excel 是誰改的？決議在哪一版？」

🔍找不到、想不起

明明看過，卻得重新翻、重新問、重新想。

「主管臨時問一個數字，我知道看過，卻找不到。」

🔁重複推導

同樣的整理、同樣的分析，半年後又從零做一次。

「新人問我這個流程，我又得從頭講一次。」

問題不是「資料太少」，而是 知識沒有被整理、累積、可被再利用。

換個角度想

一個塞滿的抽屜　vs.　一個整理過的大腦

🗄️傳統資料夾

  • 東西都「放著」，但沒人整理
  • 找資料 = 翻箱倒櫃 + 靠記性
  • 資料之間沒有關聯
  • 越積越多，越來越難用

🧠LLM-Wiki 知識庫

  • 每份資料都被讀過、摘要、連結
  • 用問問題的方式取用知識
  • 概念彼此交叉引用成網路
  • 越用越聰明，知識持續累積

差別在於——有一位不會累的助理，幫你把所有資料整理進大腦並隨時更新。

概念緣起

靈感來自 Karpathy 的「LLM Wiki」

AI 領域知名研究者 Andrej Karpathy 提出一個想法：與其每次都讓 AI 從頭讀一堆資料重新推論，不如讓它維護一份持久的知識庫。

知識應該「一次編譯、持續更新」，
而不是每次都重新推導。

👤人負責

提供原始資料，以及提問。就這兩件事。

🤖AI 負責

寫摘要、建立交叉引用、保持一致性、追蹤矛盾——所有維護工作。

這不是一套特定產品或軟體，而是一種知識管理方法——讓 LLM 維護一份能持續更新的 wiki。

原始出處（Karpathy 親筆）：gist.github.com/karpathy/442a6bf…　·　本分享將它落實為公司日常可用的做法。

它解決了什麼

傳統做法　vs.　LLM-Wiki

情境以前怎麼做用 LLM-Wiki
整理新資料自己讀、自己摘要、自己歸檔丟進去，AI 讀完寫摘要並接上相關頁面
找答案翻檔案、問同事、靠記憶直接提問，得到帶出處的答案
知識累積做完就忘，下次重來結論自動回填，越問越完整
一致性不同檔案說法打架沒人發現定期健檢，主動找出矛盾與過時
需要的工具各種系統、資料庫純文字檔即可，不需資料庫
核心內容 · 1

三層架構：原料、知識、規則

RAW

原始資料層（唯讀，永不修改）

你丟進來的東西：Excel、PDF、會議記錄、文件、圖片。原檔保留不動，永遠可回溯。

WIKI

知識層（AI 維護）

AI 編寫的筆記頁面，彼此用連結串成網路；含一個目錄入口與一份操作日誌。

規則

規範層（Skill）

一份「遊戲規則」，告訴 AI 該怎麼整理、命名、連結、檢查。一次設定，之後自動遵守。

my-llm-wiki/ ├─ raw/ ← 你丟進來的原始檔（不動） │ ├─ 2025年度策略會議.pdf │ ├─ 行銷月報Q4.xlsx │ └─ 外包需求確認信.eml ├─ wiki/ ← AI 維護的知識 │ ├─ index.md │ ├─ 年度目標.md │ ├─ 行銷預算分析.md │ └─ 外包-A專案.md └─ CLAUDE.md ← 規則（Skill）

看到資料夾就懂了：
你只碰最上面那層（丟資料、問問題）；
中間的知識整理，AI 全包。

→

定期體檢，找矛盾、孤立頁、缺漏。

三個動作循環往復，知識庫就越長越好。

📥 操作一

Ingest — 把新資料「吃」進來

  • 把檔案放進 raw/，跟 AI 說：「整理這份資料」
  • AI 讀完，理解核心概念
  • 新增或更新相關筆記頁面（一份資料可能影響多頁）
  • 更新目錄，並在日誌記下：日期、來源、改了哪些頁
  • 遇到流程圖、流程表，會原樣重繪保留在筆記裡
# 你說： 整理 raw/2025年度策略會議記錄.pdf # AI 做： ✓ 讀取來源、擷取重點 ✓ 新增 [[會議-2025策略會議]] ✓ 更新 [[年度目標]]、[[預算規劃]] ✓ 更新 index 目錄 ✓ 記入 log 日誌
💬 操作二

Query — 用問問題的方式取用知識

  • 直接用白話問，不用記檔名
  • AI 搜尋知識庫、讀相關頁，給出帶引用的答案
  • 看得到答案來自哪一頁、哪份原始資料——可追溯、可信任
  • 若答案值得保存，自動回填成新頁面
  • 下次同樣的問題，直接引用、不必重算
# 你問： 我們去年的行銷預算花在哪？成效如何？ # AI 答（節錄）： 主要投放在 A 通路（佔 60%）…… 成效見 [[行銷成效-2025]]。 出處：[[會議-2025策略會議]]、 　　　[[raw/行銷月報Q4]] # 結論已存入 [[行銷預算分析]]
🩺 操作三

Lint — 幫知識庫做健康檢查

知識庫養大之後，難免有資料打架或被遺忘。隔一陣子讓 AI 體檢一次：

⚔️矛盾／過時

兩頁說法不一致、或數字已被新資料推翻，主動標記出來。

🏝️孤立頁面

沒有被任何頁面連到的「孤兒」，提醒你補上關聯。

🔗缺漏引用

相關但還沒互連的概念，建議補上交叉引用。

讓知識庫不只變大，還能保持乾淨可信。

它長什麼樣子

每一頁筆記，都是一份純文字檔

--- title: 行銷預算分析 tags: [行銷, 預算, 2025] sources: - "[[raw/行銷月報Q4]]" - "[[會議-2025策略會議]]" updated: 2026-06-29 --- # 行銷預算分析 2025 全年投放以 A 通路為主…… 詳見 [[行銷成效-2025]] 與 [[通路比較]]。
  • 標頭：標題、標籤、來源、更新日期一目了然
  • 來源：每個結論都連回原始資料，可追溯
  • 連結：[[雙括號]] 把相關頁面串成知識網
  • 純 markdown：用記事本就能開，永遠讀得到
  • 可用 Obsidian 開啟，看到頁面間的關係圖

純文字 = 不被任何軟體綁架、可用 Git 追蹤版本、十年後依然打得開。

先回答三個疑慮

開始之前，你可能會先想問

🔒安全嗎？

資料放在本機；要上雲用 GitHub Private 私有儲存庫；高敏感資料不上雲，接資料庫只用唯讀帳號。

🎯可信嗎？

答案必須帶來源，可點回原始資料查證；重要結論人工確認——AI 草擬、人審核。

🙂難嗎？

從一個資料夾、一份文件開始，不必一次導入全部；簡易版完全不用碰 Git。

這三點後面 FAQ 還會更詳細——先放心，門檻比你想的低。

安裝方式

兩種方案：先簡單體驗，再正式上線

🟢簡易版　新手先用這個

🔵正式版　要備份再升級

想要版本控管與雲端備份時，再加這兩個。

  • ＝ 簡易版三項，再加：

# 正式版才需要：在知識庫資料夾內一次性設定 git init git add . git commit -m "初始知識庫" # 連到 GitHub 後： git push # 之後每次整理完 → commit 一次，留一個版本

💊正式版為什麼值得？

Git／GitHub 是知識庫的後悔藥：AI 萬一改壞或刪錯，隨時還原到任一版本；GitHub 再多一層異地雲端備份，換電腦也不怕丟。

＊建議從簡易版開始，體驗到價值後再升級正式版；不熟指令可請 Claude Code 代為設定。

常見問題 FAQ · 1／3

門檻、資料安全

Q.我不會寫程式，也沒用過這些工具，能用嗎？要學多久？

A. 核心只有兩個動作：丟檔案、問問題。工具點「下一步」就裝好；那幾行指令照貼即可，或直接請 Claude Code 代勞。大約十分鐘就能上手。

Q.我的機密資料會外流、被拿去訓練 AI 嗎？放 GitHub 不就公開了？

A. 資料存在你自己的電腦；要不要上雲由你決定，且 GitHub 請選 Private 私有儲存庫（不公開）。是否使用對話資料依方案與公司政策設定，最保險是高敏感資料只留本機。

Q.限閱、個資類的東西可以放進去嗎？

A. 依公司資安分級判斷。高敏感資料建議只放本機、不上 GitHub、不接雲端；接資料庫時用唯讀連線，AI 只能查不能改。

Q.萬一 AI 把我的資料整理錯、或刪掉了怎麼辦？

A. 這正是 Git／GitHub 的用途——一鍵還原到任何一個歷史版本。原始資料（raw/）也永遠保留不動，隨時可重來。

常見問題 FAQ · 2／3

信任、成本、怎麼導入

Q.AI 會不會亂編、講錯？我怎麼知道能不能相信？

A. 每個答案都帶出處，會標明來自哪一頁、哪份原始資料，你可以點回去查證——不是憑空生成。

Q.要花錢嗎？可以放多少資料？「1 份」是什麼？

A. Claude Code 需訂閱，其餘工具免費。「1 份」＝一個來源檔案（一份 PDF／Excel／會議記錄，不論頁數）。約 百來份內用單純結構就運作良好；更多也不會壞，只要分主題建子索引、或拆成多個知識庫即可，依然不需要向量資料庫。

Q.這要取代公司現有的共用磁碟 / Notion / SharePoint 嗎？

A. 不取代，是補充。差別在於它會主動讀懂、整理、交叉連結，並用帶出處的方式回答你——不只是把檔案放著。

Q.我這個職務怎麼用？回去第一步做什麼？團隊能共用嗎？

A. 挑一份你最常翻的檔案丟進去、問一個問題，就開始了。團隊可共用同一個私有儲存庫協作維護。

常見問題 FAQ · 3／3

再深入一點的疑問

Q.為什麼不直接用 RAG／向量資料庫？那不是更強嗎？

A. RAG 解決的是「上千份的語意檢索」；這套解決的是「我的幾十份工作文件被讀懂、連結、持續更新」——規模不同。它刻意走輕量：零基礎設施、純文字、可 git 追蹤。真有大規模即時檢索需求，再上 RAG 即可，兩者不衝突。

Q.AI 整理錯的內容，會不會累積、污染整個知識庫？

A. 機制上有三道防線：①答案帶出處，錯誤可即點即查、可定位；②raw/ 永不修改＋git 版本控管，可回溯、可還原；③Lint 健檢專抓矛盾與過時。定位是「AI 草擬、人審核」，不是不容質疑的真理。

Q.會不會被綁死某個 AI／廠商？模型升級會讓結果不一致嗎？

A. 不會綁死——產物是純 markdown ＋ git，換工具、換模型、換廠商都帶得走、讀得到。已寫成文字的頁面是固定的，模型升級只影響「下次怎麼整理」，不會回頭改寫舊頁。

Q.我自己很熟我的檔案，維護這套划算嗎？

A. 對個人當下，邊際效益也許不高；價值在三個未來才付代價的點：交接（你腦中的索引帶不走）、規模成長後、跨人協作。你的大腦是最好的索引——直到休假、離職或別人接手那天。

深入版 · 內外包窗口（可略）

內外包窗口：建立同一個事實來源

痛點

需求散在 Email、訊息、會議；廠商常問「這之前怎麼講的？」；版本、決議、驗收標準各說各話。

AI 整理成

  • 每個外包專案一頁：現況、決議、待辦、窗口
  • 需求變更歷史：誰、何時、改了什麼
  • 驗收清單：標準與達成進度
# 對齊交付 上週跟廠商 A 確認的交付項目與時程？ # AI：引用 [[外包-A專案]]，列項目＋日期＋出處 # 防止各說各話 這個需求變更前後差在哪？誰拍板的？ # AI：引用 [[需求變更-0612]]，標記決策人與時間 # 把雜訊變規格 把這串 email 整理成給外包的需求規格 產出本週對廠商的進度追蹤清單

對內對外都看同一份事實——少了推諉，責任清楚，交接不漏。

深入版 · 主管／PM（可略）

主管／PM：把跨專案現況一手掌握

丟進 raw/ 的東西

各專案週報、會議記錄、KPI 數據、風險議題、人力與時程狀況。

AI 整理成

  • 跨專案進度儀表：誰在做什麼、到哪了
  • 風險清單：議題、負責人、狀態
  • 決策／待辦追蹤：哪些逾期
# 掌握全局 這週各專案進度如何？有哪些紅燈項目？ # AI：彙整各週報，列進度＋亮紅燈項目＋出處 # 盯風險與待辦 還沒解決的風險有哪些？負責人是誰？ 上次決議的待辦，哪些逾期了？ # 秒出週報 把這週彙整成給老闆的一頁重點週報

從「追著各團隊要狀態」變成「直接問知識庫」——週報秒出、風險不漏接。
